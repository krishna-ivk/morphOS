# Durable Execution Handoff Contracts

This document defines the minimum contract surface between:

- `openai/symphony`
- `wavezync/durable`

for real durable execution in `morphOS v0`.

It is the concrete follow-up to:

- [morphOS v0 runtime contracts](MORPHOS_V0_RUNTIME_CONTRACTS.md)
- [Symphony and Durable authority boundary](SYMPHONY_DURABLE_AUTHORITY_BOUNDARY.md)

Those docs define the objects and the ownership split.
This doc defines the handoff messages that allow the split to operate in practice.

## Executive Summary

The minimum Durable handoff surface is:

1. `DurableExecutionRequest`
2. `DurableExecutionStatus`
3. `ExecutionCheckpointRecord`

These contracts exist to answer three runtime questions:

- what exactly is Symphony asking Durable to execute
- what exactly is Durable reporting back while execution is alive
- what exactly is the checkpoint record that makes resumption real

Without these contracts, `ExecutionCheckpoint` becomes theoretical instead of operational.

## Design Rules

### Rule 1

Symphony sends semantic intent.
Durable should not need to infer workflow meaning from opaque payloads.

### Rule 2

Durable sends runtime truth.
Symphony should not invent checkpoint or retry state in observability payloads.

### Rule 3

Every handoff object must be correlatable to:

- `run_id`
- `task_execution_id`
- `issue_identifier`

### Rule 4

Every handoff object must support persistence, replay, and operator inspection.

## 1. DurableExecutionRequest

### Purpose

`DurableExecutionRequest` is the contract Symphony sends when a workflow step needs real durable execution.

It is the message that turns:

- workflow meaning
- step intent
- retry eligibility
- approval state

into an actual durable runtime record.

### Owned By

Produced by:

- `Symphony`

Consumed by:

- `Durable`

### Minimum Fields

- `execution_request_id`
- `run_id`
- `task_execution_id`
- `issue_identifier`
- `workflow_template_id`
- `step_id`
- `step_type`
- `requested_by_agent_id`
- `execution_kind`
- `input_context_refs`
- `input_artifact_refs`
- `tool_action`
- `retry_policy`
- `timeout_policy`
- `approval_state`
- `requested_at`

### Notes On Meaning

- `execution_kind` should be explicit enough for Durable to know what class of execution to host
- `tool_action` should carry actionable external-work intent only when the step actually invokes a tool or integration
- `retry_policy` is a hint from workflow semantics, not a durable-owned attempt record
- `approval_state` tells Durable whether the execution is runnable now or must remain blocked

### What It Must Not Contain

Do not overload this request with:

- full workflow graph definition
- memory retrieval policy
- UI-only formatting data
- ad hoc checkpoint fields generated by Symphony

## 2. DurableExecutionStatus

### Purpose

`DurableExecutionStatus` is the callback or event Durable emits while execution exists.

It tells Symphony:

- whether the execution is running
- whether it has retried
- whether it is blocked
- whether it produced a checkpoint
- whether it completed or failed

### Owned By

Produced by:

- `Durable`

Consumed by:

- `Symphony`
- operator/observability surfaces

### Minimum Fields

- `execution_status_id`
- `run_id`
- `task_execution_id`
- `issue_identifier`
- `durable_execution_id`
- `status`
- `attempt`
- `checkpoint_id`
- `checkpoint_state`
- `retry_state`
- `blocking_reason`
- `failure_reason`
- `started_at`
- `updated_at`
- `ended_at`
- `result_artifact_refs`

### Allowed Status Values

- `accepted`
- `running`
- `blocked`
- `paused`
- `retry_scheduled`
- `retrying`
- `completed`
- `failed`
- `cancelled`
- `expired`

### Allowed Retry State Values

- `none`
- `eligible`
- `scheduled`
- `in_progress`
- `exhausted`

### Key Behavior

This is the contract that lets Symphony react without pretending to be the kernel.

Examples:

- if `status = blocked` and `blocking_reason = approval_pending`, Symphony can keep semantic state blocked
- if `status = retry_scheduled`, Symphony can show retry posture without owning the retry timer
- if `status = completed`, Symphony can evaluate workflow transitions

## 3. ExecutionCheckpointRecord

### Purpose

`ExecutionCheckpointRecord` is the durable-owned checkpoint object referenced by `ExecutionCheckpoint` in the shared runtime contracts.

This is the first place where checkpoint data becomes real.

### Owned By

Produced by:

- `Durable`

Referenced by:

- `Symphony`
- CLI
- Command Centre

### Minimum Fields

- `checkpoint_id`
- `run_id`
- `task_execution_id`
- `durable_execution_id`
- `status`
- `created_at`
- `updated_at`
- `resume_token_ref`
- `snapshot_artifact_ref`
- `resumable_until`
- `invalidated_at`
- `invalidated_reason`

### Allowed Status Values

- `created`
- `active`
- `restored`
- `expired`
- `invalidated`

### Key Rule

Symphony may expose checkpoint metadata.
Durable must remain the source of truth for checkpoint identity and lifecycle.

## Ownership Matrix

### Symphony Owns

- when to create a durable execution request
- the semantic meaning of the step
- whether a task is retryable in workflow terms
- how a completed durable status changes workflow progression

### Durable Owns

- durable execution record identity
- attempts
- retry scheduling
- checkpoint lifecycle
- resumability truth
- timeout and expiry truth

### Shared Consumption

- CLI
- Command Centre
- validation and audit surfaces

These layers may observe all three contracts.
They must not become the primary source of truth for any of them.

## Recommended Event Flow

1. Symphony creates `TaskExecution`
2. Symphony emits `DurableExecutionRequest`
3. Durable accepts and creates its internal execution record
4. Durable emits `DurableExecutionStatus(status=accepted)`
5. Durable emits further status updates:
   - `running`
   - `blocked`
   - `retry_scheduled`
   - `retrying`
   - `completed`
   - `failed`
6. If resumable state exists, Durable creates `ExecutionCheckpointRecord`
7. Symphony uses the status stream to update semantic workflow state
8. Operator surfaces observe both semantic state and durable state, without collapsing them

## Approval Example

For an approval-gated deploy:

1. Symphony identifies that `approve_release` blocks `deploy`
2. Symphony creates the `Directive`
3. Durable keeps the related execution in `blocked`
4. Operator issues an `ApprovalDecision`
5. Symphony applies the decision semantically
6. Durable receives the unblock signal and resumes or starts execution
7. Durable emits updated `DurableExecutionStatus`

This preserves the split cleanly:

- approval meaning stays in Symphony
- blocked/resumable execution truth stays in Durable

## Program Runner Example

For a durable `program_runner` step:

1. Symphony emits `DurableExecutionRequest` with:
   - `step_type = program_runner`
   - tool/runtime payload
   - retry and timeout hints
2. Durable persists the execution
3. Durable emits `running`
4. if interrupted, Durable creates `ExecutionCheckpointRecord`
5. on completion, Durable emits `completed` with result artifact refs
6. Symphony advances the workflow graph

## What Not To Do

Do not:

- represent retry state only as Symphony-level counters
- generate checkpoint ids in presenter code
- treat operator receipts as the canonical checkpoint store
- collapse `DurableExecutionStatus` into `WorkflowRun` too early
- let tool action completion imply durable completion without a status callback

## Recommended Mapping Into skyforce-core

The next shared-contract pass in `skyforce-core` should add:

- `DurableExecutionRequest`
- `DurableExecutionStatus`
- `ExecutionCheckpointRecord`

These should be distinct from, but linked to:

- `WorkflowRun`
- `TaskExecution`
- `ToolAction`
- `ExecutionCheckpoint`

## Recommended Implementation Order

1. define these three contracts in `morphOS`
2. mirror them into `skyforce-core` types
3. add a Durable adapter boundary in the runtime layer
4. emit real status/checkpoint data from Durable
5. only then project checkpoint state through Symphony observability
